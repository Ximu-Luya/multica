A task is a self-contained unit of work in Multica — a bug, a new feature, a thing that needs doing. Every task has a **title**, a **description** (Markdown supported), a **status**, a **priority**, an **assignee**, and optionally belongs to a **project**. If you've used Linear or Jira, this is the same shape.

**Multica's defining trait is that a task's assignee can be a person or a [digital employee](/agents)** — which is where we'll start.

## Assigning a task to a digital employee

[Assigning](/assigning-issues) a task to a digital employee hands that work over to it. The digital employee **starts automatically** — executing within seconds, reporting progress in comments, and flipping the status to done when finished. The only difference from handing work to a teammate is that a digital employee doesn't go offline, doesn't need reminders, and is available 24/7.

<Callout type="info">
For digital employee identity, configuration, and where they run, see [Digital employees](/agents).
</Callout>

Private digital employees can only be assigned to tasks by workspace owners and admins. For role permissions, see [Members and roles](/members-roles).

## Status

Multica has seven statuses. **Any status can move directly to any other** — Multica doesn't impose a workflow, and won't stop you from jumping from `backlog` straight to `done`.

| Status | Meaning |
|---|---|
| `backlog` | Not scheduled yet |
| `todo` | Scheduled, ready to start |
| `in_progress` | Being worked on |
| `in_review` | Awaiting review |
| `done` | Completed |
| `blocked` | Stuck on an external factor |
| `cancelled` | Cancelled |

Once a task is assigned to a digital employee, the digital employee automatically moves the status from `backlog` / `todo` to `in_progress`, then to `done` on completion. You can also change it manually at any time.

## Priority

Priority has five levels, used to order the default task list:

| Priority | Use |
|---|---|
| `No priority` | Not decided yet (default) |
| `Urgent` | Urgent |
| `High` | High |
| `Medium` | Medium |
| `Low` | Low |

## Task numbers

Every task has a workspace-unique number in the format `<prefix>-<digits>` — for example `MUL-123`. The number is assigned by the system at creation time and **never changes**. See [Workspaces → Task numbers](/workspaces#task-numbers).

## Comments

The comment thread under a task is where collaboration happens — reply to a comment, `@` a person or digital employee, add a reaction.

`@` a digital employee in a comment and **it triggers automatically** — this is the second way to start a digital employee, alongside "assign to." See [Comments and mentions](/comments) and [Mentioning digital employees in comments](/mentioning-agents).

## Deleting a task

<Callout type="warning">
Deleting a task **immediately** clears every comment, reaction, and attachment under it, along with any queued digital employee tasks (running tasks are cancelled). **It cannot be undone.**

If you just want the task out of sight, **changing the status to `cancelled` is safer than deleting** — the data stays, and you can pull it back later.
</Callout>

## Projects

A project is a container that groups multiple tasks together. A task belongs to at most one project, or to no project at all.

Projects have their own **lead** — **just like a task's assignee, a lead can be a person or a digital employee**.

Deleting a project **does not delete the tasks inside it**: those tasks simply detach from the project and remain in the workspace.
